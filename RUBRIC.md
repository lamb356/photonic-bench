# PhotonicBench Daily-Use Improvement Rubric

Use this rubric for checklist completion decisions and the mandatory Hostile
Senior Reviewer critique.

## Visualizer Usefulness

- Comparison mode helps a daily user answer practical questions:
  - which card is most energy-efficient;
  - which card is fastest;
  - what the throughput/latency/energy tradeoffs are;
  - which claims are local-model estimates versus published references;
  - which artifacts are calibrated, source-backed, or purely local.
- Analytics are deterministic, explainable, and visible without requiring a
  notebook.
- Mixed schema comparisons remain readable.
- The UI stays static-site friendly and generated by the CLI.
- Text, controls, and tables remain usable at desktop and mobile widths.

## Saved Preset Quality

- Presets have stable names and stable artifact references.
- Presets survive visualizer regeneration.
- Stale artifact references produce clear warnings rather than broken screens.
- Users can load presets with minimal clicks.
- Preset implementation does not require a hosted backend.

## Export Quality

- Machine-readable exports include selected artifact IDs, summary metrics,
  derived comparison metrics, provenance status, and modeling-boundary notes.
- Human-readable exports are suitable for copying into notes or reviews.
- Exports do not overstate local-model estimates as published reproduction.
- Export behavior is covered by tests or browser smoke verification.

## Published Card Quality

- Each new card is backed by a published source with enough citation detail to
  find it again.
- Paper-reported metrics are carried under published reference/provenance
  fields and kept separate from local component-model outputs.
- Any surrogate workload mapping is named explicitly.
- Assumptions record missing source details and modeling choices.
- Generated Markdown and JSON artifacts are committed/regenerated together with
  their YAML config.

## Modeling And Transformer Quality

- Formula changes are explicit, auditable, and documented.
- New JSON fields are schema-documented where applicable.
- Tests cover normal cases and validation failures.
- Transformer convenience features preserve existing decomposed-card auditability.
- The model does not silently claim device-level reproduction of source papers.

## Documentation Quality

- README examples stay runnable.
- User-facing behavior is documented near the relevant workflow.
- Model docs distinguish current formulas, exclusions, and local assumptions.
- Schema docs stay aligned with generated JSON.

## Verification Quality

- Focused tests are run near the changed areas.
- Full Ruff and pytest are run before final completion.
- Visualizer generation is run after report/artifact changes.
- Browser smoke testing verifies the new visualizer workflow.
- Generated artifacts are inspected or asserted for stale/broken content.

## Review Readiness

- Checklist DONE items include proof.
- State files are current.
- The Hostile Senior Reviewer critique is recorded in `PROGRESS.md`.
- Important critique findings are fixed before completion or explicitly
  justified.
- Remaining risks are named plainly.
